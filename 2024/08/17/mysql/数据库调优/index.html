mysql性能调优

性能瓶颈：mysql在单表带索引不做分区的情况下，可以扛得住6000w左右的数据量，各方面优化的特别好的情况下最多到8000w，再往后性能就跟不上了；所以mysql大数据量存储时如何优化？

解决思路

  • 定时删除：只保留N年内的数据
  • 冷热归档：常用热数据放在MYSQL，冷门数据放在OceanBase或其他数据库
  • 垂直切分：按不同类型的数据放在不同的数据库中
    • 分为数据垂直切分和业务垂直切分；
    • 我们都是用微服务，所以一般都是做好了业务垂直切分；
    • 数据垂直切分指的是比如一张表10个字段，把常用的5个字段放在一个表里，不常用的5个字段放在另一张表里
  • 水平切分：将相同类型数据放在多个容器（partition、table、database）中
    • 如果以上方案解决不了，就需要水平分切。就是分区|分表|分库

MYSQL分区

  • MYSQL原生分区功能 支持事务
  • 对外逻辑上表现为1张单表 底层由多个物理区块组成
  • 不需要考虑取模、分页、排序等问题
  • 支持Hash、Range、Key、List、Composite
  • 主键必须包含分区键
  • 分区表不支持外键
  • 目前公司MYSQL分区只支持时间字段并且强制设置保留分区数量
    • 不建议使用，原因是公司的mysql分区只能强制使用时间字段 而且强制保留分区数量也就是用起来保存几年的数据差不多，超过会被删掉；同时不支持外键

MYSQL分表

  • 查询和写入时需要指定sharding（取模或哈希）
  • 在同一个库中，支持事务
  • 需要注意在分页、排序等问题
  • 数据任在同一台物理机上

MYSQL分库

  • 查询和写入时需要指定sharding（取模或哈希）

  • 跨库不支持事务

  • 需要注意在分页、排序等问题

  • 数据在不同的物理机上

  • 有个缺点；数据可能在不同物理机上，如果sharding做的不好，取数可能跨好几台机器

方案选择

何时需要水平切分

  • 当性能出现瓶颈并且其他优化手段无法解决的时候
  • 注意：mysql标准配置如下，虽然可以找dba去加，但是加配置一般也加不了多少
    • cpu核数：40
    • 内存(GB): 256
    • 磁盘(GB): 215G/2.2T

如何选择分库分表

只分表

  • 单表数据量较大，单表读写性能出现瓶颈
  • 单表瓶颈的标志：单表数据量较大，导致读写性能慢，索引效果差
  • 经过评估单库的容量和性能可以支撑未来3-5年的增长

只分库

  • 数据库（读）写压力较大，数据库出现存储性能（IO）瓶颈
  • 单库瓶颈的标志
    • 一般都是硬件指标快满了，就要分库了
    • CPU压力过大（load过高）导致读写性能较慢
    • 内存不足（缓存池命中率较低）导致读写性能较慢
    • 磁盘不足（IO负载过高）导致无法正常读写数据
    • 带宽不足，导致读写性能较慢

分库分表

  • 单表数据量较大，单表读写性能出现瓶颈
  • 数据库（读）写压力较大，数据库出现存储性能瓶颈

分库分表的注意事项

  • ！！！非必要不分库分表 ！！！
  • 避免跨库（分布式）事务、避免联合查询
  • 尽量把同一组数据Hash到同一个DB服务器上
  • 谨慎处理Count、Group、分页、排序等问题（DAL）
  • Sharding数量最好是2的幂次方充分考虑未来增长（3~5年）

规范

使用规范

  • 禁止在线上生产环境做数据库压力测试
  • 禁止在数据库中明文存储密码和敏感数据
  • 禁止从开发、测试环境直连线上生产数据库
  • 禁止存储诸如图片、影像之类的二进制数据
  • 禁止将业务日志实时保存到数据库
  • 业务部门推广活动，请提前通知DBA

建表规范

  • 存储引擎必须是InnoDB
  • 必须有显示主键
  • 推荐使用独立业务的AUTO_INCREMENT列做主键，禁止使用多字段联合主键
  • 不使用UUID/MD5/HASH等函数生成的无序值做主键
  • 不允许使用外键、存储过程、视图、触发器
  • 不建议使用TEXT类型的字段
  • 字符集必须是utf8mb4
  • 字段声明NOT NULL，且必须显示指定默认值
  • 库名、表名、字段名 禁用保留字，如desc、range、match、delayed等
  • 对表和字段都添加备注说明（特别是枚举）
  • 主键列如果为整型，统一使用bigint
  • 涉及货币金额等类似精度敏感的数据必须使用decimal替代float和double
  • 库名、表名、字段名必须都是小写
  • 必须包含时间戳字段datachange_lasttime，定义默认值为on update CURRENT_TIMESTAMP
  • datachange_createtime作为行创建时间，定义默认值时请不要添加on update CURRENT_TIMESTAMP
  • 对较长的字符类型，如果需要索引，则建立前缀索引，前缀索引长度控制在20个字符以内
  • 单张表索引数量控制在5个以内，组合索引字段控制在3个以内
  • 组合索引字段的顺序，去重之后数量越多代表区分度越高，区分度越高的字段排在前面
  • 有出海需求的数据库需要增加 userdata_location（UDL） 字段

索引规范

  • 背景：组合索引 idx_a_b_c(a,b,c)

  • 原则一：避免全表扫描

  • 背景：组合索引 idx_a_b_c(a,b,c)

  • 原则二：避免排序

  • 原则三：避免回表

SQL规范

  • 禁止使用子查询
  • 禁止使用select *，必须指定需要的字段
  • update/delete/replace只能单表操作，且必须基于主键操作，如果是删除全表，建议使用truncate
  • 数据订正（update/delete）时，要先select，避免出现误删除，确认无误后再执行更新语句
  • insert语句要显示指定插入的列名，且不允许使用insert … select , replace … select 这样的bulk insert形式
  • 不允许出现多于一次的join；需要join的字段，数据类型保持绝对的一致；多表关联时，保证被关联的字段有索引
  • 不允许在查询语句中使用hint类似的语法：如 sql_no_cache, force index, ignore key, straight join等
  • 不要频繁在数据库中做运算操作，如count()/sum()/md5()/order by rand()等
  • 不要使用大偏移量的limit
    • 正例： where id > N limit M (N由程序计算出来)
    • 反例： where ？limit N，M （N越来越大）
  • in后边的集合元素的数量，尽量控制在200个以内
  • or查询索引列尽量使用in谓词
    • 正例：select * from t where col1 in (‘a’,’b’)
    • 反例：select * from t where col1 = ‘a’ or col1 = ‘b’
  • 连接MySQL不要设置成autocommit=0
  • 线上尽量少使用大查询，将复杂SQL拆分成多条简单SQL，化繁为简
  • 如果结果集不需要去重，尽量改写union 为 union all
  • ORDER BY 语句对多个不同条件使用不同方向的排序无法使用索引，请尽量避免或在程序端进行排序
  • SQL优化原则，执行计划至少达到range级别，运算复杂度由好到坏排序
    • system: O(1)
    • const: O(1)
    • eq_ref: O(log n)
    • ref: O(log n) ==> ref以上基本是利用索引完成的等值查询或更快的常量内部优化的查询
    • range: O(log n)+ ==> 利用索引完成的范围查询（建议至少达到这个级别）
    • index: O(n) ==> 全索引扫描
    • ALL: O(n) ==> 全表扫

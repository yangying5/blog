HA[高可用]

Redis架构演进

主从架构

  • 主从模式保证了数据的高可用，并且搭配了哨兵集群，可以实时的监控每个节点的状态变化，如果遇到机器故障就会主动发起主从切换，保证数据的高可用

官方redis cluster架构

  • 随着互联网的发展，数据的存储会越来越大，并发量会越来越高，对于单片[group]的redis 就支撑不下
  • 官方支持了一个redis集群的模式，但是现在使用的并不多，因为有一些限制
    • 灵活度没有自研的高
    • Redis官方出来的比较晚，大部分公司自研redis集群的时候，redis官方还没有集群的功能
    • Redis内置的哨兵模式在节点扩展方面有一定的限制，无法支持大规模的Redis集群

自研cluster架构

  • 公司自研的redis集群架构，通过key的哈希，去找到redis对应的分片进行读写
  • 内置哨兵集群，对redis集群进行监控
  • 哨兵集群存在一个或多个哨兵节点，监控着Redis集群中的主节点和从节点的状态，并在主节点出现故障时，自动将从节点升级为新的主节点，保证Redis集群的高可用性

redis集群类型

分为四种类型

  1. 单机房缓存：非核心的场景或者出海的场景会使用；他不具备机房之间的故障转移；机房内某个机器故障是可以做到恢复
  2. 本机房缓存：可以做到机房故障之后，客户端通过切流+正常的链接去连接到正常的机房上
  3. Xpipe单向同步：使用最多；master在一个机房，能保证数据写入的一致性，通过单向同步的方式把数据去同步到另一个机房
  4. Xpipe双向同步：两边同时可以读写

Xpipe单向同步 和 Xpipe双向同步的区别

  • Xpipe是Redis提供的一种数据同步的解决方案，可以实现Redis集群之间的数据同步和实时备份。Xpipe有两种不同的同步方式：单向同步和双向同步。
    • Xpipe单向同步：指的是从源Redis节点将数据同步到目标Redis节点，实现源节点数据的备份或者实时同步。在单向同步中，数据只能从源节点同步到目标节点，目标节点不会主动将数据同步回源节点。这种方式适用于主从节点的数据同步和备份。
    • Xpipe双向同步：指的是源节点和目标节点之间的数据可以相互同步，实现数据的双向同步和备份。在双向同步中，源节点和目标节点都可以作为主节点，相互之间可以进行数据同步和备份。这种方式适用于多节点之间的数据同步和备份。
  • 总的来说，Xpipe单向同步适用于主从节点之间的数据同步和备份，而Xpipe双向同步适用于多节点之间的数据同步和备份。另外，Xpipe双向同步需要注意数据一致性问题，需要进行数据冲突的解决和数据同步的优化。

数据类型适用场景

过期删除策略

  • 主线程：惰性删除：同步删除和异步删除，同步会阻塞；公司默认是异步
  • 后台线程：定期删除：定期到过期字典里找key删除

内存淘汰策略

  • 策略分为三类

    • noeviction策略：没有淘汰，比较危险，当redis满了，再去写入会报错
    • volatile策略：对有ttl过期时间的淘汰策略
    • allkey策略：对所有key都会淘汰

  • 淘汰算法：

    • TTL：如果有设置过期时间，针对过期时间的删除
    • Random：随机找key去删除
    • LRU算法：目前使用最多的，基于”最近最少使用”原则的缓存淘汰策略，也是公司默认的配置
    • LFU：最不频繁使用

redis最佳实践

Key/Value设计

  • Key可读性和可管理性、控制长度、不包含特殊字符
  • Value不要过大(集合类型控制元素个数)
    • （标准：String < 100KB，List/Hash/Set/ZSet < 1万个元素）
  • 设置合理TTL
  • 缓存信息精简：DB/接口获取的结果转换后再写入

热key危害

  • 热key：比如做秒杀活动，对同一个商品id就会产生热key的访问；这个时候key的数量一般都是几个，一般会录入到固定的redis分片里，这个时候就会导致某一个分片的请求数量过多，导致cpu过高，产生不稳定

热key如何发现


针对热key，首先发现热key：
1. 访问量比较高就有问题，所以可以埋点去看每个group的访问量，通过脚本去找到redis访问量比较高的group，就认为她很有可能产生热key的问题
2. 通过dba服务号去抓取热key的操作，找到具体的group后

常见问题与解决方案

限流: 可以采用redis实现分布式的限流，保证系统的稳定
分布式锁，不同的用户抢同一个资源时产生锁竞争，单机的话可以用java提供的解决；分布式下需要引入组件比如redis实现分布式锁，minos是基于哈希实现的分布式锁，基于哈希的好处是可以把当前的线程id保存到我们的哈希表里。然后下一次同一个线程去获取锁的时候 可以实现可重入；如果用其他类型没办法实现可重入

穿透：比如商品id传入很大的值或负数，他在redis和db并不存在
击穿：redis没有，db有；缓存的某一个key过期，正好遇到这个key的热点，大量的流量就会到db，db压力就会大
雪崩：完全依赖db，db压力大

在我们提前知道全量数据的情况下，从过滤器判断是否存在这个值
击穿：限制并发数，类似限流的效果，放一定量的请求到db，超过就拒绝或等待
过期时间加随机数，让批量过期不会同时触发

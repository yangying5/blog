基于JavaAgent探针技术的慢SQL治理实践

慢查询的现状

企业内部的现状

  • 大部分数据库是共享集群,一个实例下会有多个DB, 是为了节省成本; 给每个db都做一个独立的集群，费用会非常高；
  • 共享集群问题：集群内一个实例打满资源会影响其他业务
  • 集团2022发生了多起慢查询引起的故障
  • 针对慢查询的治理较为滞后
  • 无法有效处理低频的慢SQL(DBA记录慢SQL对次数和频率有要求)
  • SQL的规范较依赖研发个人自觉;无法有效约束

DBA目前如何处理

  • 慢查询出现 & 并发50，触发循环Killer
    • 随机kill正常请求和慢查询请求
    • 这里应改为随机kill慢查询请求，否则会误伤正常请求
  • 滞后触发可能无法成功接入
    • 因为是滞后的操作，如果QPS高，在killer没有介入前，机器可能就被打挂了
    • 这里是否被打挂取决于killer介入的阈值，接入的时机可以取决于：
      • 机器剩余资源的阈值
      • 单个慢sql 持续的时间

SRE如何处理

全生命周期治理方案

  • 设计阶段(堵源头)：
    • 添加SQL人工评审环节，加载SQL规范宣讲
    • 未来方向：SQL评审上系统评审，通过系统输入SQL来检测SQL是否满足规范
  • 研发阶段(禁新增)：检测不合理的SQL并进行告警阻断(按级别)
  • 生产阶段(减存量)：
    • 对不合理的SQL进行告警(按级别)
    • 对耗时异常增长的SQL进行告警(此时慢SQL还没发生)
  • 故障阶段(控影响)：发生故障时主动Kill掉慢查询，减少同集群其他业务影响

设置完超时，jdbc会做什么呢

  1. 查询的同时创建一个定时任务
  2. 定时任务被调度时查询还没完成则发起KILL
  3. 查询完成时取消定时任务

Java Agent

  • Java Agent Instrument 是一种能够在编译后运行时修改字节码的技术

Java Agent：正常CLass加载过程

Java Agent：增强class加载过程

Java Agent：静态模式

java -javaagent:myAgent.jar -javaagent:myAgent2.jar -jar App.jar

Java Agent: 动态模式

  • 99.99%的选择：动态模式
  • 步骤：

    VirtualMachine vm = VirtualMachine.attach(“目标程序的pid”); # 拿到目标程序的VM
    vm.loadAgent(“myAgent-1.0.jar”); # 把Agent发送到目标VM

Java Agent: 字节码增强的过程

  • 最后都会进入VM_Operation, 这里会暂停jvm的运行
  • 注意：因为会暂停jvm的运行，所以使用agent时要考虑它的性能，高QPS还是要慎用
  • agent对资源的消耗占比情况：有埋点，也就是agent接和不接 在资源消耗上和性能方面有什么区别，结论是：做了以上几个步骤，整体的耗时不会超过1毫秒；大多数场景都是可接受

Luna组件

Agent主要功能

  • 动态设置超时
  • 主动kill超时的查询
  • 方法级别的耗时埋点
  • 驱动层拦截SQL

Service主要功能

  • 分析采集到的SQL
  • 查询SQL执行计划
  • 对不合理SQL进行告警
  • 对耗时异常SQL进行预警
  • 对全表扫描SQL进行告警
  • 检测索引并提示优化
  • 对发生Kill的SQL进行告警

动态设置超时

  • 要根据业务场景设置超时，比如核心业务，订单服务，那目标应该是尽全力保证这条sql被执行到
  • 负载高，超时时间设置低一点；负载很低，那就是有大把的资源，超时时间设置高一点；能尽量保证业务被执行完
  • 负载采集对实时性要求高，比如高qps场景，采集慢，很有可能适得其反；也就是实际上资源已经被打满了，采集的慢，可能我们得到的结论是还有资源

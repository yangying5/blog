dump

  • 根据手机是arm还是 arm64，所以选择 memdumper文件 push 到测试环境上。
  • 打开样例正常执行，仅 dump 目标 SO 所处的内存范围
  • 运行报错在so所处内存外的地址，继续dump成bin文件
  • 遇到了一个新的内存错误 0x6f5d895d10

    Read memory failed: address=0x6f5d895d10, size=8, value=0x0

  • 这说明 lpAppInfo 并不是保存在 SO 地址范围里的，而是通过 malloc 等函数做的内存分配，用 memdumper64 dump 对应地址 0x1000 字节长度的内存

    ./memdumper64 -p com.aikucun.akapp -m -s 0x6f5d895d10 -e 0x6f5d896d10 -n 6f5d895d10.bin -o /sdcard

  1. 手动将SO 对应的内存写到 Unidbg 虚拟环境
  2. 遇到so内存以内的地址报错，大概率是libc.so报错，可以直接patch
    样本想要访问外部的 strlen 函数，我们是 dump 对应的 libc 内存，还是怎么做？脸哥认为走原生的执行流（即 dump 和加载 libc 的内存环境）或自己 patch 比较好，我更青睐于使用 Unidbg 加载内部的 libc，做一下嫁接。
  3. 遇到so内存以外的地址报错，用MemDumper工具dump成bin文

花指令扫盲

花指令是由设计者特别构思，希望使反汇编出错，让破解者无法清楚正确的理解程序逻辑的一种技术
企图隐藏掉不想被逆向工程的代码块 (或其它功能) 的一种方法, 在真实代码中插入一些垃圾代码的同时还保证原有程序的正确执行, 而程序无法很好地反编译, 难以理解程序内容, 达到混淆视听的效果.

花指令的意图

  • 花指令试图去干扰 IDA、Ghidra 这些分析工具，让反汇编过程出错
  • 有的花指令未必得迷惑分析工具，恶心分析人员也是可行的
  • 指令膨胀 增加了我们分析的“体量
  • 实际场景中我们遇到的花指令主要会干扰工具，因为干扰工具的花指令往往也能干扰分析人员，一举两得。

区分花指令

比如 Windows 上的“热补丁”技术，会将函数开头预留几个字节的空间，填充为 nop。这看起来和 P1 是类似的，但它是花指令吗，当然不是。它的意图是为 Inline Hook 提供便利。

常见花指令特征

  1. JNI_OnLoad 作为一个鲜明的入口，就可以称得上关键函数。对它的反编译是否正常和顺利，往往就能判断出是否有花指令或其他代码保护手段
  2. 栈开的也太小了，如只有 32 字节，它存进去了四个 X 系列寄存器，每个占 8 字节空间，直接满了，难道程序没有局部变量要处理？
  3. 备份的寄存器很奇怪，X1、X2、X3 没有备份的必要，这些本就是函数中需要被使用到的、易变的寄存器，它们的值被破坏是可预见以及可接受的。
  4. 只有函数序言没有函数尾声

汇编语言之序言和尾声

  • 函数开头的几行代码叫函数序言，与之对应，函数结尾处的几行代码叫函数尾声。

  • 函数序言： 会修改 SP 寄存器，让它的值更小，因为栈默认是从高地址往低地址方向增长，所以这是开辟栈空间的动作，一般称为开栈。新开辟的栈内存用于存储局部变量以及不想被污染的寄存器。局部变量不必多说，不想被污染的寄存器指的是在当前函数中这些寄存器可能被使用，进而其值会被改变，但我们希望在函数结束后可以恢复这些寄存器，这可以理解为备份

  • 函数尾声：函数序言的逆操作，比如我们举例的这个函数，将不想被污染的寄存器 完完整整的从栈中取出，即恢复它们为进入函数前的状态，然后修改 SP 寄存器，同样回到原先状态，和函数序言形成对应，开多少栈就清多少，即清栈。

自动化或半自动化去除花指令

简易方案

  • junk code 的核心特征是“事倍功半——较多的代码量只对寄存器和内存环境做了微小甚至单一的改变。
  • 那么如果我们定位到花指令的起始处，我们称此处是初始状态（包括寄存器、堆栈、内存），然后使用指令模拟器往下执行，在某一位置，运行状态相较于初始状态仅有极小的差异，比如仅有单个通用寄存器的值不同，那么此处大概率就是 junk code 的结束位置

激进方案

  • 就是不用指定起始执行地址，工具采用随机或遍历执行的方式，寻找可以“压缩”的执行片段。
  • 比如限定每次执行至少十行以上汇编，如果两端的寄存器状态差异极小，那么就认为是花指令的边界，然后生成对应的等效代码并 patch处理。
  • 这个方案很激进，好处当然很大，根本不需要分析人员去理解花指令以及抽象出对应的匹配特征。但缺点也很大，存在很大的不稳定性以及可能有很严重的误判。
  • 比如 ida-pro-junk-code-removal 、nao 这些项目虽然很简陋，其实都有这种思路的影子
